| 维度 | 概况 |

## 二、产品亮点（做得好的地方）

### ⭐ 架构决策清醒

**AI与规则引擎分离**是该产品最正确的架构决策。从 V2.0 将评分从 AI 黑盒中剥离，改为「AI 负责信息提取 → 规则引擎负责评分」，解决了 AI 评分不一致、无法解释、无法调优的核心痛点。评分配置通过 `industry_config.json` 和 `country_weights.json` 外置，支持业务人员零代码调整 — 这是成熟产品的思维。

### ⭐ 迭代节奏务实

从 CHANGELOG 可以清晰看到「发现问题 → 出改进文档 → 落地实现」的闭环。V2.2 改进文档提出的「客户跟进状态管理」和「抓取失败可视化」两大方向，都在 V2.2.1 落地。改进文档质量高（需求文档、错误文档、发现页修改方案等 7 份），说明作者有产品思维，不只是堆功能。

### ⭐ 三层缓存设计合理

搜索缓存(30天) + 官网缓存(7天) + AI分析缓存(内容哈希)，有效控制 API 费用。对于 SerpAPI 250次/月、Tavily 1000次/月的免费配额限制，这个设计直接决定了产品是否可日常使用。

### ⭐ 断点续跑 + 局部重试

搜索任务的断点续跑（`current_keyword_index`）+ 客户级别的重新抓取/重新分析 API，避免了「一步失败全盘重来」的糟糕体验。这是一个容易被忽视但用户感知极强的设计。

### ⭐ 多语言搜索

60+ 国家的本地语言映射，AI 自动翻译+扩展关键词。对于非英语国家的外贸开发（如沙特、波兰、墨西哥），这是决定性功能 — 没有它，搜出来的都是美国公司。

## 三、核心问题（按严重程度排列）

### 🔴 P0：routes.py 严重膨胀，已成单体巨石

`app/api/routes.py` 单文件约 **1068 行**，包含了客户管理、搜索发现、跟进状态、数据同步、Excel 导出等所有 API。这是典型的「一开始快，后面越来越慢」的技术债。

**危害：**
- 维护成本线性增长，每次改一个功能都要在千行文件中定位
- 无法独立测试某一组 API
- 所有接口共享同一个 `_analyzing_set` 和 `_running_tasks` 全局状态，耦合度极高
- 新人接手成本巨大

**建议：** 拆分为独立的路由模块（按 V1/V2 版本号拆分已无意义）：

```
api/
  customers.py      # 客户CRUD + 分析 + 跟进状态
  discovery.py       # 搜索发现任务
  sync.py            # 数据同步
  export.py          # Excel导出
```

### 🔴 P0：前端轮询架构已到瓶颈

改进文档「发现页修改方案V1」已经准确地指出了问题：`setInterval` 每 3 秒全量 `innerHTML` 重建 DOM，多个轮询可能同时运行，无 `AbortController` 超时保护。当搜索任务涉及 50-200 个关键词时，这些问题的叠加效应会非常明显。

**当前改进文档中提出的方案（SSE/WebSocket）方向正确，但尚未实施。**

### 🟡 P1：`_global_stop_flag` 全局变量存在并发隐患

```python
_global_stop_flag = False  # 模块级全局变量
```

当前设计意味着**同时只能运行一个搜索任务**。如果产品将来需要支持多用户（甚至同一用户多任务并行），这个全局标志会直接导致 A 任务停止 B 任务。

**建议：** 改为字典映射 `{task_id: stop_flag}`，每个任务独立控制。

### 🟡 P1：测试覆盖不均衡

129 个测试覆盖了纯逻辑模块（评分引擎、去重、邮箱提取、关键词分析等），这很好。但**零集成测试** — 没有任何一个测试走完整的「搜索 → 抓取 → 分析 → 评分」链路，也没有 API 层面的测试（FastAPI TestClient）。这意味着核心流程的正确性完全依赖手工验证。

**建议：** 至少为以下场景添加集成测试：
- 客户导入 → 分析 → 评分 → 导出 完整链路
- API 层面的 CRUD 测试（使用 TestClient）

### 🟡 P1：配置文件无 Schema 校验

```python
config_path = os.path.join(os.path.dirname(__file__), "industry_config.json")
```

配置文件没有 JSON Schema 校验，如果用户手动编辑时写错一个字段名（如 `wastewater` 拼成 `wastewate`），系统不会报错，只是静默地少加分 — 这种 bug 极难排查。

**建议：** 启动时校验配置文件结构，对缺失字段给出明确警告。

### 🟠 P2：产品存在「水处理行业锁定」

`industry_config.json` 的正向关键词全部围绕水处理行业（wastewater, sewage, biogas, desalination…）。虽然配置文件可以修改，但评分引擎的 `_score_project_match` 硬编码了 `has_water_content` 逻辑。如果用户想切换到光伏或包装机械行业，需要改代码而非纯配置。

**建议：** 将行业特定的检测关键词也外置到配置文件中。

### 🟠 P2：`错误文档.txt` 记录了已知但已修复的问题

`错误文档.txt` 是 1732 行的运行时错误日志（`no such column: customers.status`）。虽然 V2.2.2 的自动迁移机制（`_migrate_add_column`）已经解决了这个问题，但错误日志文件留在项目中作为「文档」是一个危险的信号 — 可能还有其他未覆盖的迁移场景。

## 四、产品策略建议

### 🎯 短期（V2.8，1-2天）

| 优先级 | 任务 | 预期效果 |
|--------|------|----------|
| P0 | **拆分 `routes.py`** | 当前投资回报率最高的改进，直接影响后续所有功能的开发效率 |
| P0 | **前端网络请求加 AbortController + timeout** | 改进文档已写好方案，直接实施即可 |
| P1 | **JSON Schema 校验配置文件** | 5行代码 + 一个 schema 文件，杜绝静默配置错误 |
| P2 | **删除或归档 `错误文档.txt`** | 已知且已修复的错误不要存为「文档」，改为在 CHANGELOG 中记录 |

### 🎯 中期（V3.0，1-2周）

| 优先级 | 任务 | 预期效果 |
|--------|------|----------|
| P0 | **SSE 替代轮询** | FastAPI 原生支持 `EventSourceResponse`，改动量可控，体验提升巨大 |
| P1 | **任务队列化** | 将搜索任务从 `asyncio.create_task` 迁移到后台任务队列，解决全局标志并发问题 |
| P1 | **行业配置解耦** | 将水处理特定的评分逻辑抽象为行业插件，让光伏、包装、建材等行业可零代码切换 |
| P1 | **API 集成测试** | 至少覆盖核心流程的 happy path（FastAPI TestClient） |

### 🎯 长期（V4.0+，战略方向）

| 方向 | 说明 |
|------|------|
| **多用户支持** | 加入用户认证，从单机工具升级为团队 SaaS。PostgreSQL 支持已为此铺路 |
| **邮件自动化** | 从「发现 → 分析 → 跟进记录」延伸到「发现 → 分析 → 一键发送开发信 → 追踪打开率」 |
| **Embedding 语义匹配** | V2.5 相似客户扩展已是规则模型，引入向量相似度可大幅提升精准度（改进文档中已规划） |
| **Chrome 插件** | 业务员浏览 LinkedIn/Google 时一键导入公司，降低录入摩擦 |

## 六、一句话总结

> 这是一个**产品思维远超平均水平的独立开发者作品**。核心架构决策（AI 与规则分离、配置驱动评分、三层缓存、断点续跑）体现了成熟的产品判断力。当前最大的瓶颈不是功能缺失，而是代码组织（routes.py 单体化）和架构（全局变量、轮询）已经开始拖累迭代速度。建议用 1-2 天做一次结构化重构，为 V3.0 的多行业、多用户扩展打好基础。
