# Phase 1：多源瀑布式邮箱发现升级方向

## 背景与目标

当前项目的邮箱发现能力依赖单一的 Hunter.io，每月仅有 25 次免费额度，容易在月中耗尽。Phase 1 的目标是通过引入 Tomba 作为第二数据源，并增加自研抓取兜底，构建一套**瀑布式（Waterfall）邮箱发现机制**，在不增加成本的前提下显著提升邮箱发现成功率。

---

## 核心思路：瀑布式调用

瀑布式的核心原则是**只有上一级无结果时才触发下一级**，从而最大化利用各平台的免费额度，避免浪费。

```
输入：公司域名
       ↓
[第1级] Hunter.io domain search
       ↓ 无结果或结果数量不足
[第2级] Tomba domain search
       ↓ 仍无结果
[第3级] 自研网页抓取兜底（从官网提取 mailto: 链接）
       ↓
结果合并 → 去重 → 验证 → 评分排序 → 输出
```

---

## 各数据源说明

### Hunter.io（现有，保留）
- 免费额度：25 次搜索 + 验证/月（共享）
- 现状：已集成，继续作为第一优先级
- 优化方向：将其调用结果纳入统一返回格式，与后续源对齐

### Tomba（新增，核心补充）
- 免费额度：注册送 75 credits，之后每月 25 次搜索 + 50 次验证
- API文档：https://docs.tomba.io/api/finder#domain-search
- 关键优势：
  - **无结果不扣费**：查询某域名若没有邮箱数据，本次不消耗 quota，可以放心尝试
  - **验证免费捆绑**：验证额度独立于搜索额度，不互相消耗
  - 同一域名 30 天内重复查询只计一次
- 使用方式：REST API，需要 `X-Tomba-Key` 和 `X-Tomba-Secret` 两个认证头
- 配置方式：在 `.env` 中新增 `TOMBA_API_KEY` 和 `TOMBA_SECRET`

### 自研抓取兜底（新增，零成本）
- 适用场景：Hunter 和 Tomba 均无数据的小众公司/小语种站点
- 实现方式：请求公司官网 HTML，用正则提取 `mailto:` 链接
- 注意事项：
  - 只提取明文邮箱，不做 JS 渲染（成本低、速度快）
  - 结果置信度较低，需标记来源为 `scraped`，评分权重低于 API 来源
  - 注意过滤掉通用邮箱（如 info@、support@、noreply@）

---

## 结果处理逻辑

### 合并与去重
- 三个来源的结果统一为同一数据结构（email、name、position、source、confidence_score）
- 以邮箱地址为 key 去重，同一邮箱取置信度最高的那条记录

### 验证优先级
- Tomba 返回的邮箱自带验证状态，优先复用，不额外消耗 Hunter 验证额度
- Hunter 返回结果中已包含验证状态，同样直接使用
- 自研抓取的邮箱无验证状态，标记为 `unverified`

### 评分排序
结合现有评分引擎，对发现的邮箱按以下维度排序输出：

| 维度 | 权重说明 |
|------|---------|
| 来源 | API来源 > 自研抓取 |
| 验证状态 | valid > unknown > unverified |
| 职位级别 | 决策人（CEO/采购/Sales）> 普通员工 > 通用邮箱 |
| 置信度分数 | Tomba/Hunter 返回的原始 score 字段 |

---

## Quota 管理原则

- 各平台的已用次数应**持久化存储到 SQLite**，而非保存在内存中（当前 Hunter quota 计数器重启后会重置，这是已知 bug，Phase 1 一并修复）
- 每次调用前先检查剩余 quota，不足时跳过该源，降级到下一级
- 建议在数据库中记录每次调用的域名、来源、结果数量、时间戳，便于后续分析哪个源命中率更高

## 配置变更

`.env` 新增以下字段：

```
# Tomba
TOMBA_API_KEY=ta_xxxxxxxxxx
TOMBA_SECRET=ts_xxxxxxxxxx

# 瀑布行为控制
EMAIL_DISCOVERY_MIN_RESULTS=2        # 结果数低于此值才触发下一级
EMAIL_DISCOVERY_ENABLE_SCRAPING=true # 是否启用自研抓取兜底
```

## 预期收益

| 指标 | 改进前 | 改进后（预估） |
|------|-------|-------------|
| 月可查询域名数 | ~25 | ~50（Hunter+Tomba叠加） |
| 邮箱发现成功率 | 取决于Hunter覆盖率 | 提升，尤其对非英语区域公司 |
| 验证额度消耗 | Hunter搜索+验证共享25次 | Tomba验证独立，不抢搜索quota |
| 成本 | $0 | $0（均在免费tier内） |

## 与后续阶段的衔接

Phase 1 完成后，邮箱发现模块将以统一接口对外暴露，Phase 2 的 Pipeline 分层重构可以直接将其作为 `EmailDiscovery` Step 接入，无需再做改动。
