## 🚨 一、Bug 类（优先修复）

### 1. `customers.py` — `analyze_single` 官网抓取失败后无声返回

**问题位置**: `app/api/customers.py:232-269`

**现象**:  
`scrape_website` 返回空字符串时，函数跳过了 `if website_text:` 块，既不设置 `analyzed_at` 也不返回任何响应。客户状态停留在"未分析"，API 返回 `null`（HTTP 200）。

**对比**: `search_task_service.py` 中 `_auto_analyze_and_save` 在同样场景正确设置了 `analyzed_at` 和 `scrape_status = "failed"`。

**修复建议**:
```python
if website_text:
    # ... 现有分析逻辑
else:
    customer.scrape_status = "failed"
    customer.fail_reason = "官网抓取失败"
customer.analyzed_at = datetime.now(timezone.utc)  # 无论成败都设置
db.commit()
return {"message": "分析完成", "customer_id": customer.id, "website_text": bool(website_text)}
```

### 2. 全局定时任务状态 — ASGI 多 Worker 不兼容

**问题位置**:  
- `customers.py:26` — `_analyzing_set: Set[int]`  
- `search_task_service.py:39-40` — `_task_stop_flags` / `_batch_stop_flag`

**现象**:  
三个模块级全局变量在单进程下工作正常，但如果是 Gunicorn/Uvicorn 多 Worker 启动或在多进程下使用，各进程状态不同步，导致：
- 停止信号无法传递到另一个 worker
- 分析状态显示异常
- `len(_analyzing_set) > 0` 检查失效

**修复建议**:  
用数据库行（`UPDATE search_tasks SET stop_requested = 1 WHERE id = ?`）或 Redis 存储任务状态。

### 3. `search_task_service.py` 内部重复 import

**问题位置**: `app/services/search_task_service.py:33` 和 `:296`

顶层已导入 `from app.services.deduplication import find_existing_customer`，`_auto_analyze_and_save` 函数内部又再次 `from ... import ...`。虽不影响运行但不符合规范。

### 4. `waterfall_discovery.py` — `hunter_entries` 判断逻辑

**问题位置**: `app/services/waterfall_discovery.py:412`

```python
if len(hunter_entries) < MIN_RESULTS:  # 默认 MIN_RESULTS = 2
```
当 MIN_RESULTS 配置为 2 时，Hunter 返回 1 条也会触发 Tomba。但如果用户期望精确 2 条才跳过，该逻辑正确。只是需要确认这个阈值是否合适。

## 🧱 二、架构与代码质量

### 5. `datetime.utcnow()` 废弃迁移

**影响范围**: 全项目 40+ 处

Python 3.12+ 中 `datetime.datetime.utcnow()` 已标记为 deprecated，3.14+ 将移除。

**统一替换方案**:
```python
from datetime import datetime, timezone
now = datetime.now(timezone.utc)
```
涉及文件: `database.py`、`customers.py`、`discovery.py`、`search_task_service.py`、`excel_importer.py`、`sync.py`、`cache_manager.py`、`hunter_service.py`、`tomba_service.py`、`waterfall_discovery.py` 等。

### 6. 三处重复的 `_extract_domain` 函数

| 文件 | 行号 |
|------|------|
| `app/api/hunter.py` | 29-37 |
| `app/api/tomba.py` | 21-28 |
| `app/services/waterfall_discovery.py` | 41-49 |

三个实现逻辑相同（去除协议 → 取第一段 → 去端口和参数）。应统一收归到 `app/services/url_normalizer.py`。

### 7. 日志系统不统一

整个项目混合使用两种日志方式：

| 方式 | 数量 | 示例 |
|------|------|------|
| `print()` | ~50+ | 各处散落 |
| `logging.getLogger()` | ~6 个模块 | `search_task_service.py`、`hunter_service.py` 等 |

**问题**:  
- `print()` 无法控制日志级别、格式、输出目标
- 各模块 `logger` 各自独立，无统一配置
- 中英文日志信息混合

**建议**: 在 `main.py` 或 `app/__init__.py` 配置统一的 logging 配置：
```python
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(name)s:%(levelname)s] %(message)s",
)
```
并逐步将 `print()` 替换为 `logger.info/warning/error`。

### 8. API 路由间代码重复

**客户列表** (`customers.py:44`) vs **发现结果列表** (`discovery.py:282`) 的分页、筛选逻辑高度重复，可提取为公共函数。

`hunter.py:229-255` 和 `tomba.py:221-250` 的 `usage`、`clear-cache`、`cache-entries` 端点模式高度相似，可考虑抽象缓存管理基类。

### 9. 缺少 Pydantic 请求/响应模型

当前所有 API 参数用 `Query(...)` 注入，返回用 `dict` 手拼。项目规模扩大到 30+ 端点后，建议引入 Pydantic 的 `BaseModel`：
- 自动请求校验（现在手动 parse + raise 400）
- 自动响应文档生成（OpenAPI schema）
- IDE 类型提示

### 10. `site_classifier.py` 废弃模块

**问题位置**: `app/services/site_classifier.py`

整个文件中：
- `classify_domain()` — 站点分类函数  
- `build_priority_paths()` — 语言路径字典  
- `RedirectTracker` — 重定向跟踪类

**没有任何文件 import 它**，是 V2.3 遗留但未集成的模块。建议决策：删除或集成入 `website_scraper.py`。

## ⚡ 三、性能优化

### 11. 批量分析串行执行

**问题位置**: `customers.py:279-294`

```python
for customer in customers:
    await analyze_single(customer.id, db)  # 串行
```

**建议**: 改用 `asyncio.gather` + 信号量控制并发：
```python
async def analyze_all(db: Session):
    customers = db.query(Customer).filter(...).all()
    semaphore = asyncio.Semaphore(3)  # 最大 3 并发
    async def _analyze_one(c):
        async with semaphore:
            return await analyze_single(c.id, db)
    results = await asyncio.gather(*[ _analyze_one(c) for c in customers ])
```

### 12. Excel 导出全量加载

**问题位置**: `customers.py:317-398`

```python
customers = db.query(Customer).order_by(...).all()  # 全量内存加载
```

当客户数达到 5000+，内存占用和锁表时间明显增加。建议：
- 使用数据库游标（`yield_per`）分块读取
- 或使用 `StreamingResponse` + 临时文件

### 13. AI 分析 prompt 截断丢失关键内容

**问题位置**: `deepseek_analyzer.py:23`

```python
truncated_text = website_text[:3000]
```

对于有大量项目案例的大网站，仅截取前 3000 字符可能完全错过关键信息。

**建议**: 智能段落选择——从全文计算关键词密度，优先保留高密度段落：
```python
# 伪代码思路
paragraphs = website_text.split('\n\n')
keyword_score = [sum(kw in p.lower() for kw in POSITIVE_KWS) for p in paragraphs]
top_paragraphs = [p for _, p in sorted(zip(keyword_score, paragraphs), reverse=True)[:5]]
prompt_text = '\n'.join(top_paragraphs)[:4000]
```

### 14. 公司名模糊查询无法使用索引

**问题位置**: `deduplication.py:162-177`

```python
Customer.company_name.ilike(f"%{min_token}%")  # 前后通配符 → 全表扫描
```

随着客户数增长，每次导入新客户都触发 O(n) 扫描。建议：
- 对 `company_name` 添加 SQLite FTS5 全文索引 或 PostgreSQL GIN 索引
- 或存储标准化公司名字段，在该字段上建索引

### 15. SSE 端点每次轮询创建新 DB Session

**问题位置**: `discovery.py:214`

```python
while True:
    db = SessionLocal()  # 每秒创建/关闭
    try:
        task = db.query(SearchTask)...
    finally:
        db.close()
    await asyncio.sleep(1)
```

建议复用任务开始时传入的 session，或在连接建立后打开一个 session 持续使用。

## 🎯 四、功能缺失

### 16. 瀑布式邮箱发现 — 缺少前端 UI

**状态**: 后端 100% 完成（`waterfall.py` + `waterfall_discovery.py` 共 450+ 行），但客户详情页尚未集成瀑布式查找的 UI。

**当前入口**: 仅可通过 `GET /api/waterfall/email-discovery?website=xxx` 调用

**建议实现**: 在详情页增加「多源查邮箱」Tab，展示三级的查询日志和合并结果：

```
┌─ 跟进状态 | Hunter查邮箱 | 🌊多源查邮箱 ──┐
│                                            │
│  域名: xxxxx.com                            │
│  [开始瀑布式查找]                           │
│                                            │
│  查询日志:                                  │
│  ✓ 第1级 Hunter 查找: 0 条                 │
│  → 第2级 Tomba 查找: 5 条                  │
│  ═ 共 5 个邮箱（含来源标注）               │
└────────────────────────────────────────────┘
```

### 17. 相似客户扩展 — 无法一键导入

**问题位置**: `similar_company_finder.py`

当前 `find_similar_companies()` 返回相似客户列表但只展示，不能一键入库。

**建议**: 在返回结果中添加「导入选中客户到列表」按钮，后端增加批量导入端点。

### 18. 搜索词扩展 — 无手动确认

**问题位置**: `discovery.py:37-46`

调用 `POST /api/discovery/expand-keywords?keyword=xxx` 后直接开始搜索，用户无法编辑扩展结果。

**建议**: 拆分为两步：
1. 先扩展 → 返回关键词列表供用户编辑/确认
2. 用户点击确认后再提交搜索任务

### 19. 瀑布式邮箱发现配置 — 无批量重跑界面

对历史客户重新执行瀑布式发现：当前无入口，只能单个客户在详情页触发。

### 20. 数据库中没有索引加速常用查询

例如 `discovery_source + country` 联合查询在所有发现相关 API 中使用频繁，但没有联合索引。

## 🔒 五、安全与健壮性

### 21. 官网抓取 SSL 验证默认关闭

**问题位置**: `website_scraper.py:17`

```python
_VERIFY_SSL = os.environ.get("SCRAPE_VERIFY_SSL", "").lower() == "true"
```

默认 `verify=False`，HTTPS 请求不验证证书，存在中间人攻击风险。建议默认开启，加入已知证书问题域名白名单。

### 22. `analyze_single` 抛出原始异常

**问题位置**: `customers.py:271`

```python
except Exception as e:
    db.rollback()
    raise e  # 原始异常 -> 可能暴露内部细节
```

建议包装为 `HTTPException`：
```python
except Exception as e:
    db.rollback()
    raise HTTPException(status_code=500, detail=f"分析失败: {str(e)[:100]}")
```

### 23. 批量删除使用 Query 参数传数组

**问题位置**: `customers.py:414-434`

```python
@router.post("/customers/batch-delete")
def batch_delete_customers(
    ids: str = Query(..., description="要删除的客户ID列表，JSON数组字符串"),
):
```

URL Query 参数传递 JSON 数组字符串：
- 长度受限（浏览器/nginx 对 URL 长度限制）
- 不符合 RESTful 设计

**建议**: 改用 Request Body：
```python
class BatchDeleteRequest(BaseModel):
    ids: List[int]

@router.delete("/customers/batch")
def batch_delete(data: BatchDeleteRequest, db: Session = Depends(get_db)):
```

### 24. CSV 注入风险

导出 Excel 时，`ai_summary` 字段中包含用户/AI 生成文本，若包含 `=HYPERLINK()` 等公式开头，在其他工具（如 Google Sheets）中打开时可能触发 CSV 注入。

## 📦 六、工程化改进

### 25. 完整锁定依赖版本

`requirements.txt` 中写死了 8 个依赖的精确版本，但未锁定间接依赖。建议：
```bash
pip freeze > requirements.lock  # 或使用 poetry/uv
```

### 26. 缺少类型注解

项目规模已达 ~5500 行 Python 代码，但函数签名基本没有类型注解（仅 `retry_manager.py` 使用了 `TypeVar`）。

建议逐步添加，优先在 API 路由和服务层：
```python
def normalize_url(raw_url: str) -> str: ...
def calculate_scores(website_text: str, ...) -> Dict[str, Any]: ...
```

### 27. 默认 `reload=False` 影响开发体验

`main.py:83` 硬编码 `reload=False`，每次改代码需手动重启。建议：
```python
uvicorn.run(
    "main:app",
    host="127.0.0.1",
    port=8000,
    reload=os.environ.get("DEV_MODE", "").lower() == "true",
)
```

### 28. 缺少代码格式化/静态检查配置

无 `.pre-commit-config.yaml`、`pyproject.toml`、`.ruff.toml` 等配置。建议引入：
- **Ruff** — 极速 Linter + Formatter（替代 flake8 + isort + black）
- **mypy** — 可选静态类型检查

### 29. 缺失 CI/CD 配置

无 GitHub Actions 或其他 CI 配置，无法自动运行测试和检查。

### 30. README 重复行

`README.md:21-22` 有两行完全相同的描述：
```
| **网页配置编辑器** | 通过网页修改行业关键词... |  # 第21行
| **网页配置编辑器** | 通过网页修改行业关键词... |  # 第22行（重复）
```
